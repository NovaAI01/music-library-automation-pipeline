# Test Coverage Map

This map explains the current collected pytest suite from the saved collection
output and direct inspection of the test sources.

Collection command used for this map:

```bash
python -m pytest --collect-only -q
```

Collected count: 637 tests.

This is a collection count and a regression coverage map. It is not a claim of
exhaustive correctness for every catalog, private library layout, metadata
source, Docker runtime, CI environment, or remediation decision.

The suite is collected from `tests/` and `tools/portfolio_demo/tests/`. Some
collected tests come from parametrized functions, so one function can produce
multiple collected nodeids. The largest example is
`tests/test_golden_entity_resolution.py`, where one parametrized golden-case
function expands across fixture CSV rows.

Safety boundaries remain part of the evidence: metadata-only validation where
documented, no audio downloads, no local media mutation, no canonical graph
mutation in public validation manifests, and no external service dependency
unless a test explicitly verifies URL construction or a mocked fetch path.

## Practical Coverage Areas

- scanner and local evidence capture
- database, ledger, and run-manifest behavior
- identity, classification, alias, and filename parsing
- placement planning and execution boundaries
- duplicate detection, review, quarantine, and restore
- metadata audit, planning, suggestions, review decisions, and knowledge
- metadata-only external ingestion and validation
- artist-credit and release-identity analysis
- source quality comparison reports from existing validation runs
- canonical graph, confidence, boundaries, roles, and governance
- UI/report views and runtime health endpoint
- public fixture workflow boundaries
- portfolio demo tooling
